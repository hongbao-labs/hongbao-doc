# 安全模型

Hongbao 的安全建立在一个原则之上：**用密码学和公开合约取代信任，而不是依赖承诺。** Hongbao 公司不会持有任何私钥，也不会接触发卡方或持卡人的资金；整个流程都在链上运行，资金始终由发卡方自己掌控。

本页汇总了 Hongbao 的产品安全模型。对于持卡人的实际体验，请见 [持卡人常见问题](receiver/faq.md)；对于发卡方的操作与合规问题，请见 [发卡方常见问题](issuer/faq.md)。

---

## 四道独立防线

下面的每一道防线，都是单独成立的、足以阻止卡上资产被盗：

1. **私钥永远不会离开芯片。** 出厂时，私钥由芯片内的硬件 TRNG 随机生成，并被物理永久封存。无论是制卡厂、发卡方、Hongbao，还是持卡人本人，都无法将其读出。
2. **必须通过物理按键授权。** 蓝牙劫持并不能完成后续操作：没有按键确认，就没有签名。连接时需要长按 3 秒，签名授权时需要长按 10 秒。
3. **收款地址写入签名内容。** 持卡人输入的地址是签名内容的一部分；一旦被篡改，签名会立即失效，合约也会直接拒收。任何 Relayer 或中间人都无法把资金转向其他地址。
4. **卡片只能签一次。** 签名完成后，固件会拒绝再次签名，合约也会将该卡标记为已领取。领过的卡只剩下纪念意义，里面再没有可提取的资产。

## 合约层保障

- **无 owner、无 admin、无暂停、无升级。** 所有特权操作都被刻意移除。合约中只保留两个核心动作：deposit（发卡方锁定资金）和过期赎回（锁定期结束后，发卡方收回未领取的资金）。
- **过期前，发卡方无法收回可领取的资金。** 资金一旦被锁定，礼物就无法被作废——这是红包的根本承诺，由代码强制执行。
- **最小锁定期限为 30 天。** `MIN_LOCK_TIME = 30 天` 是硬编码常量，确保持卡人有足够时间领取。
- **部署后即不可变。** 每个 Pool 只部署一次，随后不会再修改。新特性会以新版本合约的形式发布；已有 Pool 继续按部署时的逻辑运行，已锁定资产不会受影响。

## 任务份额的资金安全

任务卡会将资金拆成基础份额和任务份额。即便某个任务解锁凭证（预映像）泄露，**也无法被冒领**：任务资金会被强制流向持卡人首次签名时锁定的地址，并且每个凭证都绑定特定的链、Pool、卡和任务槽，跨链、跨卡、跨任务都无法重放。

## 硬件层保障

- secp256k1 私钥由芯片内硬件 TRNG 生成，并被物理永久封存
- MCU 与安全芯片之间采用 HMAC 绑定，防止换芯片攻击
- SWD 烧录被锁定，防止固件被提取
- 工厂锁定，防止出厂后被植入恶意指令
- 安全芯片具备防侧信道攻击和防物理拆解的设计

## 可用性保障：如果 Hongbao 不存在了怎么办？

**无论 Hongbao 处于何种状态，资产在合约层始终是安全的。** 合约部署在公链上，没有 owner，也不依赖任何中心化服务。领取能力由协议本身保证：

- 常态下，持卡人通过官方 Web / App 领取；短期故障通常会自动恢复。
- 作为兜底，合约 ABI、BLE 接口规范和 CLI 工具均已开源——任何具备技术能力的人都可以基于公开接口直接领取；不会操作的用户，也可以联系我们的技术支持团队。

**任何情况下，资产都不会被永久卡死在合约里。**

## 审计与披露

合约已**完全公开**于 [github.com/hongbao-labs/contracts](https://github.com/hongbao-labs/contracts)，欢迎任何人自审，或委托独立审计。团队本身具备安全背景，合约由资深审计员设计并完成内部评审。第三方审计已列入计划，具体时间将取决于订单规模和客户需求。审计报告完成后，将在此处同步披露。

## 漏洞披露

如发现漏洞、密码学问题或合约风险，请邮件至 **security@hongbao.digital**。在公开披露前，请为我们预留合理的响应窗口。所有联系方式请见 [contact.md](contact.md)。
