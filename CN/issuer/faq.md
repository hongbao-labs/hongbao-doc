# 发卡方常见问题

## 关于合约

**Q：合约有 owner 吗？我能控制吗？**
没有 owner，没有 pause，没有 admin，没有升级。我们刻意去掉了所有特权操作——这同时是去信任的设计、也是最低合规摩擦的设计。你能做的只有 deposit 和过期赎回。

**Q：能不能把过期时间设短一点，比如 7 天？**
不能。`MIN_LOCK_TIME = 30 天` 是合约硬编码常量。原因：给持卡人足够时间领取，避免发卡方和持卡人因短锁定期产生扯皮。如果你的活动场景非要短期锁定，可以 fork 合约自己改——但那就不是我们的标准合约了，需自行审计。

**Q：能让我做特权赎回吗（不等过期就把没领的钱拿回）？**
不能。如果你能提前赎回，就意味着你随时能让用户的"礼物"作废，这违背了红包的根本承诺。如果你需要这种能力，请重新评估是否适合用 Hongbao。

**Q：合约审计了吗？**
还没有第三方审计。团队自身是安全背景，合约由资深审计员设计与自审。当前团队判断不值得为早期阶段付审计成本；后续按订单规模/客户要求决定是否引入第三方背书。代码完全公开 [TBD: GitHub URL]，欢迎自审或独立审计。

**Q：合约会扣手续费吗？**
不会。链上零手续费、零抽成，刻意如此——最小化合规摩擦。Hongbao 的收入来自硬件销售、项目方定制服务费、以及 App 内与钱包 / 交易所等生态的合作分成，**全部在协议层之外**，与发卡方的资金面解耦。

**Q：合约会怎么升级？**
**不升级**。Pool 是一次性部署、不可改的。如果未来发现问题或需要新特性，会发新版本合约（新 Factory），老的 pool 继续按部署时的逻辑运行，已 deposit 资产不受影响。

## 关于流程

**Q：approve / batchDeposit / 部署 Pool 这些链上操作我自己要写脚本吗？**
不需要。Web Dapp 已经把 Factory 部署、Pool 创建、approve、批量 deposit、批量 withdrawExpired 全部封装在按钮背后——你连接 deposit 钱包、上传批次 JSON、设参数、点 Lock 就行。如果你的团队希望直接走合约层 / 自建 issuer 客户端，参考 [dev/](../dev/)。

**Q：一笔交易能发多少张？**
不需要你关心。Web Dapp 会按目标链 gas limit 自动拆分批次，每笔进度实时回显，失败可重试。底层约束是单笔 batch 受 gas limit 限制（[TBD: 各链典型 batch size]）。

**Q：能不能一次给一张卡锁多种资产？**
不能。一个 Pool 绑定一种资产。多种资产 = 多个 Pool，但每张卡每个 Pool 都能各 deposit 一次。

**注意**：不建议这么做——卡只能签一次。一张卡如果在多个 pool 都锁了资产，用户领取时只能选其中一个 pool 提领，其他 pool 的资产就永远卡在里面，只能等过期赎回。

**Q：一张卡能在多条链上各锁一份资产吗？**
技术上可以，但**不推荐**。卡只能签一次——用户在哪条链上领，其他链的资产就锁死了，只能过期赎回。除非你接受这种"互斥多链锁仓"作为某种活动机制，否则别这么用。

**Q：我可以多次给同一张卡 ERC20 续充吗？**
可以。在 Web Dapp 同一批次里再做一次 deposit 即可，过期时间以首次为准（合约层面忽略续充传入的新 lockTime）。

**Q：多个 depositor 给同一张卡充值这种高级用法支持吗？**
合约层支持（"开放模式" Pool），Web Dapp 当前不直接暴露这条路径。需要的开发者团队走合约层，详见 [dev/](../dev/)。

**Q：用户没领的钱我多久能拿回来？**
deposit 时设置的 `lockTime` 之后（最少 30 天）。在 Web Dapp 一键 Withdraw Expired，资产立刻回到你的钱包。

**Q：用户领过的卡能不能再充值？**
不能。卡领过一次后链上已标记完成，合约会拒绝再 deposit；卡本身固件层也拒绝再签名。

## 关于 App 与 Relayer

**Q：必须用 Hongbao 的 App 吗？**
是的。Hongbao 官方 App / Web 是唯一的标准持卡人入口——硬件、客户端、合约三者紧耦合，统一固件审计、蓝牙协议和 UX 是产品的安全与体验保障，也是 Hongbao 与生态伙伴（钱包 / 交易所 / Dapp）合作的入口。需要品牌化展示的项目方走**卡面联名定制**（保留 Hongbao logo，其余区域开放，详见 [customization.md](customization.md)），不通过 App 端做定制。

**Q：必须用你们的 Relayer 吗？**
不必须。`withdraw(unlockAddress, to, v, r, s)` 没有 `msg.sender` 限制，任何 EOA 都能提交。可以用我们的（默认）、自己运营、或者让用户的 App 直接用其他钱包提交（前提是用户/你出 gas）。

**Q：用你们的 Relayer 收钱吗？**
默认我们替你出 gas，目前不收费——这是早期推广，未来可能加 fair-use 政策。如果你的发放规模大、要 SLA，建议自托管 Relayer。具体见 [dev/relayer.md](../dev/relayer.md)。

**Q：能不能让用户自己出 gas？**
合约层不限制谁来提交（`withdraw` 没有 `msg.sender` 检查）。Hongbao 官方 App 默认走 Relayer 代付——这是标准持卡人体验。如果你的团队基于合约 ABI 自建客户端，可以让用户连自己的钱包付 gas（详见 [dev/](../dev/)），但用户体验会差一截，要求用户钱包里有 native 币。

## 关于损耗与售后

**Q：卡到货坏了怎么办？**
按合作合同里的 DOA 政策处理（一卡一议）。请收货时按抽样比例做验收。

**Q：用户丢卡了，能找回里面的钱吗？**
找不回（在过期前）。卡 = 私钥，丢卡 = 丢私钥 = 资产无法转出。但过期后你可以 `withdrawExpired` 把资产赎回，给用户补一张新卡。**这是我们建议的丢卡处理 SOP**：用户报损 → 你登记 → 等过期赎回 → 重发新卡 + 重新 deposit。

**Q：能否给设备序号 / 卡片防伪溯源？**
设备的 ETH 地址即唯一序列号（每张卡独立）。我们没有提供额外的防伪追踪服务——如果有需求，可以基于 ETH 地址自建 SKU 系统，定制方案 [TBD]。

**Q：卡的电池寿命？**
设计寿命约 3 年（深度休眠下基本不耗电；激活使用阶段消耗极低）。电池不可更换。

## 关于法务/合规

**Q：合规是谁的责任？**
**你的责任**。我们提供的是去信任的硬件 + 合约工具——具体派发动作（你给谁发了多少资产）由你执行，相关税务、KYC、AML、地区合规义务由你承担。我们的合约不收手续费、不留 admin key、不留任何后门，刻意减少自身的合规接触面。

**Q：你们会保留用户领取记录吗？**
我们的 Relayer 会有最低限度的运营日志（用于 abuse prevention 和故障排查），但不会留存超出必要的用户信息。如果你介意这部分日志，请自托管 Relayer。

**Q：如果 Hongbao 服务出问题（服务器宕机 / 被攻击 / 官方 App 下线 / Relayer 故障），我已经发出去的卡还能领吗？**
能。合约部署在公链上，没有 owner、不依赖任何中心化服务。卡的固件已烧录锁死，蓝牙 BLE 接口是公开规范。即便 Hongbao 的服务全部中断，你 / 用户都能：
- 用任意工具读卡（BLE 接口公开）
- 用任意 EOA 提交 withdraw（合约不限制提交者）
- 自己出 gas（不依赖任何 relayer）
