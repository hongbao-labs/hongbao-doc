# 把链上激励，变成一场只有真实用户能参与的线下任务活动

**物理礼品卡作身份门票，完成任务解锁份额，真实用户数据回流。机器人拿不到卡，更完不成任务。**

Hongbao 让项目方用一张实体卡组织线下增长活动：把代币装进卡里，设一组任务（关注、转推、进群、链上交互等）。用户拿到卡，扫码领取基础份额，完成任务解锁更多。这张卡本身就是反女巫的身份门票。活动结束，你拿到的不只是"发出去多少"，还有谁参与了、他们在链上是什么样子。

> 简单说，Hongbao 把线上 quest 平台的玩法搬到了线下，并用物理卡解决了这类平台最大的痛点：女巫。

## 你做增长活动，最头疼的是什么？

### 1. 线上任务平台，机器人比真人多

你在 quest 平台办活动，花钱买"完成任务的用户"，但其中大部分是脚本农场：

- Sybil 攻击者拿走了某些大型空投 48% 的 token（[Cointelegraph 报道](https://cointelegraph.com/news/token-airdrops-targeted-farm-accounts-sybil-attacks)）
- 行业调研显示，airdrop "eligible" 钱包里 70% 是假的（Cookie3）
- 一个项目往往要靠 IP 聚类、钱包图谱过滤几十万个 sybil 地址，才能勉强清场（[Sybil 综述](https://integral.xyz/blog/sybil-attacks)）

线上任务怎么设计，都拦不住一个人控制一万个钱包这件事。

### 2. 发完币，你根本不知道用户是谁

空投或兑换码发出去，你拿到的只是一串地址。不知道他是不是真人、活不活跃、还玩过哪些协议，没有后续触达手段，用户画像也无从沉淀。Web3 项目方烧在获客上的钱占企业价值的 29-95%，Web2 只有 7-15%（[Tokens as CAC](https://tomtunguz.com/tokens-as-cac/)）。钱花了，数据没留下。

### 3. 想做线下活动，却没有趁手的工具

会议、Meetup、地推是触达真实用户最高效的场景，但你手上没有合适的工具。纸片卡和兑换码有一堆物理风险：晒单泄露、内部盗窃、邮寄拦截（[某 YouTuber 直播时暴露 seed phrase 损失 $60K](https://decrypt.co/154522/youtuber-loses-60k-worth-crypto-after-showing-seed-phrase-stream)、[Home Depot 内部盗卡 $4M](https://www.cbsnews.com/atlanta/news/ex-home-depot-employee-sentenced-to-prison-for-stealing-over-4-million-in-gift-card-scheme-officials-say/)）。发出去的卡还会大量沉没，美国零售礼品卡每年有 $21B 没人领（[Capital One Shopping](https://capitaloneshopping.com/research/gift-card-statistics/)）。而且线下行为和线上身份对不上，活动效果没法量化。

## Hongbao 怎么做

### 一张卡 = 一个真实身份

每张卡内置安全芯片，私钥在芯片内自生成、永不导出，对应唯一的链上身份。机器人能批量注册一万个钱包，但没法批量持有一万张实体卡，更没法替你到现场完成线下任务。物理卡天然把"一个人 = 一个身份"钉死，这是线上任务平台再怎么做风控都达不到的。

### 基础份额 + 任务解锁

卡里的代币分两部分：

```
卡片总额 = 基础份额  +  任务份额（1~255 个任务，每个独立金额）
```

1. 领基础份额：用户扫码、按一次卡上的按键，基础份额到账。这一次签名同时把收款地址锁定在卡上。
2. 完成任务解锁更多：用户每完成一个任务（关注、转推、进群、链上活跃度认证等），解锁对应的任务份额，资金自动打到刚才锁定的地址。
3. 任务凭证由你掌控：每个任务的解锁凭证由你生成，可以托管在 Hongbao Web，也可以放自己后台。完成判定权在你手里。
4. 没领完的一键收回：活动结束，未完成任务的份额一键赎回回你的钱包。

> 解锁凭证就算泄露也不会被冒领，资金只会打到用户第一次签名时锁定的地址。详细机制见开源仓库（https://github.com/hongbao-labs/contracts）。

### 真实用户数据回流

用户做任务时，App 引导他绑定社交账号，链上地址则由签名绑定。你在 Web 看板能看到：

- 谁参与了：每张卡的领取状态、完成了哪些任务
- 链上画像：用户地址、gas 消耗、DEX 交互等行为，都是公开链上数据
- 实时追踪、数据导出、一键 AI 生成分析文档

数据按参与度分层。用户完成任务才解锁详细画像；不做任务的，我们只拿到一个链上地址，不强制绑定真实社交身份。隐私边界由[隐私政策](https://hongbao.digital/#/privacy)约定。

## 你能做的 / 不能做的

你能做的：

- 给任意 secp256k1 链上的资产发卡（当前 EVM 首发，含 ERC20 + ERC721；其他生态按订单扩展，详见 [README](../README.md)）
- 把份额拆成基础和任务两部分，自由设定任务内容、数量（最多 255 个）和每个任务的金额
- 设置过期时间（最少 30 天），过期未领的份额一键赎回
- 卡面联名定制（保留 Hongbao logo，其余区域开放给你的视觉设计，详见 [customization.md](customization.md)）
- 自己掌控任务完成判定（Hongbao 托管或自建后台）

你不能做的（设计上的硬约束）：

- 改变用户第一次签名锁定的收款地址，资产去向由持卡人完全控制
- 在过期前撤回用户已经可以领取的资产
- 冒领用户的任务份额，资金强制流向锁定地址

## 工作流

```
1. 下单（数量 / 目标链 / 资产 / 卡面定制方案）
2. 设计活动：基础份额 + 任务列表（每个任务金额 + 完成条件）
3. 我们寄出实体卡 + 一份 JSON（含每张卡的链上地址 + 批次元数据）
4. 项目方收货验证（抽样或全量比对地址）
5. 在 Web Dapp 一键锁定资产 + 写入任务承诺
6. 线下派发实体卡给用户
7. 用户扫码领基础份额 → 完成任务 → 解锁任务份额
8. 看板实时追踪参与与画像；过期未领份额一键赎回
```

链上动作（Factory / Pool 部署、approve、batchDepositWithTasks、claimTask、withdrawExpired）全部由 Web Dapp 封装在按钮背后，不需要写脚本。详细流程见 [端到端指南](guide.md)；开发者团队参考开源仓库（https://github.com/hongbao-labs/contracts）。

## 商业模式

Hongbao 链上零费用，合约层和 Relayer 都不收手续费。你的成本只有硬件和资产本身。我们的收入在协议层之外：

| 收入线 | 说明 |
|---|---|
| **硬件销售** | 卡片售价。B 端批量定价 + C 端零售 |
| **平台服务费** | 任务系统、数据看板、卡面联名定制、新链 / 新资产生态扩展、集成支持 |
| **App 端生态合作** | 持卡人在 App 内对接钱包、交易所、Dapp 时产生的引流分成 |

现阶段主要客户是 Web3 项目方、公链生态、交易所、钱包，典型场景包括线下会议派发、KOL 礼品、早鸟激励、社区 OG 奖励、新用户 onboarding。

## 安全保证

- 基础份额去信任：纯设备签名领取，不需要任何中心化服务参与
- 任务份额资金安全：解锁凭证就算泄露也无法冒领，资金强制流向用户首次签名锁定的地址
- 私钥从未存在 Hongbao 任何系统中：每张卡的私钥由芯片自身在出厂时生成、物理永封
- 未领取资产可赎回：过期后发卡方一键收回，资产不会卡死

合约完全公开，欢迎独立审计或自审。具体合作条款、责任划分和双方义务详见正式合同。
