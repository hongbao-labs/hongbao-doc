# Hongbao Overview

**Hongbao**，即"红包"的拼音。

在中国传统文化中，红包是过年、婚嫁、生辰里长辈递给晚辈、亲友彼此赠予的那只红色封套——里面装着钱，但承载的远不止是钱。红色辟邪纳福，封套传递祝福，**递出去那一刻，钱就是收红包人的，不需要任何第三方批准**。它是中国人对"送钱"这件事最朴素也最优雅的设计：一个物理动作完成所有权转移，社交意义和价值同时抵达。

Hongbao 协议把这套范式搬到了链上。一张实体卡就是一只红包——卡内的安全芯片在出厂时随机生成 secp256k1 私钥并物理永封，对应的链上地址收下你存入的资产；递出这张卡，资产的控制权就完整地、不可逆地转移给了拿到卡的人。**持卡即所有，签名即领取。** 没有发卡公司账户、没有激活码、没有冻结风险——所有信任假设都收敛到密码学和公开合约。**协议本身在公链上自洽运行，独立于任何中心化服务的状态。**

这就是为什么它叫 Hongbao：**用密码学还原递红包那一刻的简单与确定**。

## 我们是什么、不是什么

Hongbao 是一种**链上资产分发协议**——表面上像礼品卡、像硬件钱包，**底层是一种新的资产分发范式**。

| | 我们不是 | 我们是 |
|---|---|---|
| 对比硬件钱包 | 卖给最终用户长期管理私钥的多次签名设备 | 卖给资产分发方的**一次性签名设备**——卡递到收件人手里完成所有权转移即作废 |
| 对比传统加密礼品卡 | 平台托管资金、可冻结、可作废 | **资产锁在公开合约里**，无 owner、无 admin、不可暂停 |
| 对比传统空投 | 大量份额被女巫薅走、形成抛压、不可追回 | 物理筛选机器人、强制完成钱包注册、未领取资产可赎回 |

三个核心承诺：

- **持卡即所有**：私钥在芯片内由硬件 TRNG 生成、物理永封，无人可读出。卡递到谁手里，资产就属于谁。
- **签名即领取**：收款地址写进签名内容，任何中间人都无法重定向。卡只能签一次，签完物理作废。这是设计承诺，不是技术限制——每张卡保证一次确定的所有权转移，像现金一样不可挪用。
- **零信任设计**：合约无 owner、无 admin、无升级。Hongbao 公司不接触任何资产、不持有任何私钥，部署完合约后立即退出资金链路。

> 协议自洽：合约部署在公链上，BLE 接口和 CLI 工具开源，卡的领取能力由协议本身保证，独立于任何中心化服务的运营状态。日常体验由官方 Web/App 把关；遇到任何问题，技术支持团队随时介入。

## 不只是分发，更是任务激励

除了"递出去就到账"，Hongbao 的卡还可以是一张任务卡：份额拆成基础份额和任务份额。用户扫码领基础份额，同时锁定收款地址；之后每完成一个任务（关注、转推、进群、链上活跃度认证等），解锁一份任务份额。

物理卡天然是反女巫的身份门票。机器人能批量注册钱包，但没法批量持有实体卡，更没法替你到线下完成任务。这让 Hongbao 成为一种线下任务增长工具：真实的人、真实的参与、可以留存的用户数据。面向发卡方的完整介绍见 [issuer/](issuer/)。

## 协议组成

Hongbao 由四个组件协同构成：

| 组件 | 角色 | 形态 |
|---|---|---|
| **硬件签名设备**（红包卡） | 一次性 secp256k1 签名设备。私钥在芯片内自身随机生成，物理永封 | 由 Hongbao 设计、生产、销售 |
| **客户端**（Web / Android / iOS） | 引导持卡人完成扫码、蓝牙连接、收款地址输入、签名授权 | 由 Hongbao 官方提供 |
| **标准合约** | 链上资产托管层。绑定卡片地址，校验签名后放款；过期由发卡方赎回 | 开源，任何人可部署 / fork / 扩展 |
| **Relayer** | 代付 gas 提交 withdraw 交易 | 非信任角色，任何 EOA 都可担任 |

**链上部分（标准合约 + 签名规范）完全开放**——这是 Hongbao 作为生态协议的根基，钱包、交易所、Dapp 都可以基于这套规范集成或扩展。

**硬件与官方客户端由 Hongbao 统一提供**——硬件统一保证安全芯片来源可信、固件审计一致；客户端统一持卡人体验，也是 Hongbao 与生态伙伴（钱包 / 交易所 / 项目方）合作的入口。卡面定制（保留 Hongbao logo，其余区域开放给项目方）走商务流程，详见 [issuer/](issuer/)。

## 怎么做到的

每张卡内置一颗 tamper-resistant 安全芯片，出厂时由芯片自身随机生成 secp256k1 私钥，物理永久封存。**包括我们在内，没有任何一方能读出这把私钥。** 私钥对应的链上地址会被印成二维码刻在卡面上。

发卡方把资产锁定到这个地址（通过协议合约托管），把实体卡寄给收件人。收件人扫码连接蓝牙、输入收款地址、按下卡上按键授权签名，资产就转入收件人指定的钱包。卡片只能签一次，签完物理作废。

任务卡多一层：那一次签名领取基础份额，同时把收款地址锁定在卡上。之后用户每完成一个任务，凭任务凭证解锁对应份额，资金自动流向已锁定的地址。任务凭证即便泄露也无法被冒领，因为资金只会去那个锁定地址。合约机制详见开源仓库（[TBD: GitHub]）。

中间任何环节——蓝牙链路、网络通道、代付 gas 的 relayer——都**无法**修改收款地址、双花资产、或在过期前侵占资产。这些约束写在密码学和公开合约里，不是任何一方的口头承诺。

## 一张卡，多链通吃

Hongbao 的硬件层基于 secp256k1——加密世界事实上的通用曲线。这把所有主流公链都囊括了进来：

| 链 | 兼容方式 |
|---|---|
| **Bitcoin**（含 Taproot）、**Ethereum 及全部 EVM 链**、**Tron**、**Cosmos / Tendermint** 系、**Litecoin / Dogecoin / BCH** | 原生 secp256k1 账户曲线 |
| **Solana** | 原生 `secp256k1 program` precompile + `secp256k1_recover` syscall |
| **Sui** | 原生支持 secp256k1 账户曲线（与 Ed25519 并列） |
| **Aptos** | AIP-49 起原生支持 secp256k1 ECDSA 交易认证器 + stdlib 验签模块 |

卡片本身是 chain-agnostic 的——它对 32 字节摘要做 ECDSA 签名，不关心摘要代表哪条链上的什么交易。要支持新链，只需要在那条链上部署一份对应的"锁仓-签名验证-放款"合约。卡可以原样使用。

**当前已发布**：EVM 全系（ERC20 + ERC721 标准合约）。
**按客户需求开放**：Bitcoin、Solana、Sui、Aptos、Cosmos 等其他生态。
