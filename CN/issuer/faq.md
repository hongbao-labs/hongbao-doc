# 常见问题

## 关于合约

**Q：合约有 owner 吗？我能控制吗？**
没有 owner，没有 pause，没有 admin，没有升级，所有特权操作都去掉了。这是去信任设计：合约一经部署就自行运行，资金完全由发卡方掌控，Hongbao 不会接触。你能做的只有 deposit 和过期赎回。

**Q：能不能把过期时间设短一点，比如 7 天？**
不能。`MIN_LOCK_TIME = 30 天` 是合约硬编码常量。原因是给持卡人足够时间领取，避免发卡方和持卡人因短锁定期产生扯皮。如果你的活动场景非要短期锁定，可以 fork 合约自己改——但那就不是我们的标准合约了，需自行审计。

**Q：能让我做特权赎回吗（不等过期就把没领的钱拿回）？**
不能。如果你能提前赎回，就意味着你随时能让用户的“礼物”作废，这违背了红包的根本承诺。如果你需要这种能力，请重新评估是否适合用 Hongbao。

**Q：合约审计了吗？**
还没有第三方审计。团队本身有安全背景，合约由资深审计员设计并内部评审。第三方审计已在计划中，具体时间取决于订单规模和客户需求，报告就绪后会在此公布。代码完全公开 [github.com/hongbao-labs/contracts](https://github.com/hongbao-labs/contracts)，欢迎自审或委托独立审计。

**Q：合约会扣手续费吗？**
不会。链上零手续费、零抽成，这是刻意为之。Hongbao 不接触发卡方资金，整个流程在链上，资金由发卡方自己掌控。我们的收入来自硬件销售、项目方定制服务费，以及 App 内与钱包、交易所等生态的合作分成，全部在协议层之外，和发卡方的资金完全解耦。

**Q：合约会怎么升级？**
不升级。Pool 一次性部署、不可改。如果未来发现问题或需要新特性，会发新版本合约（新 Factory），老的 Pool 继续按部署时的逻辑运行，已 deposit 资产不受影响。

## 关于流程

**Q：approve / batchDeposit / 部署 Pool 这些链上操作我自己要写脚本吗？**
不需要。Web Dapp 已经把 Factory 部署、Pool 创建、approve、批量 deposit、批量 withdrawExpired 全部封装在按钮背后——你连接 deposit 钱包、上传批次 JSON、设参数、点 Lock 就行。如果你的团队希望直接走合约层 / 自建 issuer 客户端，参考[开源仓库](https://github.com/hongbao-labs/contracts)。

**Q：一笔交易能发多少张？**
不需要你关心。Web Dapp 会按目标链 gas limit 自动拆分批次，每笔进度实时回显，失败可重试。底层约束是单笔 batch 受目标链 gas limit 限制，Web Dapp 会按链自动适配批次大小。

**Q：能不能一次给一张卡锁多种资产？**
不能。一个 Pool 绑定一种资产。多种资产 = 多个 Pool，但每张卡每个 Pool 都能各 deposit 一次。

> 不建议这么做——卡只能签一次。一张卡如果在多个 Pool 都锁了资产，用户领取时只能选其中一个 Pool 提领，其他 Pool 的资产就永远卡在里面，只能等过期赎回。

**Q：一张卡能在多条链上各锁一份资产吗？**
技术上可以，但不推荐。卡只能签一次——用户在哪条链上领，其他链的资产就锁死了，只能过期赎回。除非你接受这种“互斥多链锁仓”作为某种活动机制，否则别这么用。

**Q：我可以多次给同一张卡 ERC20 续充吗？**
可以。在 Web Dapp 同一批次里再做一次 deposit 即可，过期时间以首次为准（合约层面忽略续充传入的新 lockTime）。

**Q：多个 depositor 给同一张卡充值这种高级用法支持吗？**
合约层支持（“开放模式” Pool），Web Dapp 当前不直接暴露这条路径。需要的开发者团队走合约层，详见[开源仓库](https://github.com/hongbao-labs/contracts)。

**Q：用户没领的钱我多久能拿回来？**
deposit 时设置的 `lockTime` 到期后（最少 30 天）。在 Web Dapp 点 Withdraw Expired，资产立刻回到你的钱包。

**Q：用户领过的卡能不能再充值？**
普通卡：不能，领过一次后链上已标记完成。任务卡：续充只进基础份额（任务槽创建后不可变），且要求卡尚未绑定收款地址、未过期、未 close。

## 关于任务与数据

**Q：任务卡是怎么运作的？**
卡里的份额拆成「基础份额 + 任务份额」。用户扫码、按一次键领基础份额，这次签名同时把收款地址锁定在卡上；之后每完成一个任务解锁一份任务份额，资金自动流向已锁定的地址。任务最多 255 个，每个可以设独立金额。

**Q：任务完成由谁判定？**
由你掌控。每个任务的解锁凭证（预映像）由你生成，可以托管在 Hongbao Web 由我们代发，也可以导出到你自己的后台自行发放，完成判定权在你手里。

**Q：解锁凭证泄露了会不会被冒领？**
不会。任务份额的资金强制流向用户首次签名时锁定的地址。凭证就算泄露，提交后钱也只会进原持卡人的地址。另外，每个凭证绑定了链、Pool、卡、任务槽，跨链、跨卡、跨任务都无法重放。

**Q：用户没完成的任务份额怎么办？**
过期后你一键 `withdrawExpired` 收回全部剩余（未领基础份额 + 未解锁任务份额）并 close 卡片。已经被用户领走的部分不受影响。

**Q：你们给我回流哪些用户数据？怎么收集？**
主要是公开链上数据：用户地址、gas 消耗、DEX 交互等行为，通过链上数据分析得到。Web 看板实时追踪、支持导出、可一键生成 AI 分析报告。

**Q：数据隐私怎么处理？**
分层处理。用户完成任务才会绑定社交账号、解锁详细画像；不做任务则只有一个链上地址，不强制绑定真实社交身份。隐私边界由[隐私政策](https://hongbao.digital/#/privacy)约定。

> 任务系统与数据看板仍在建设中，能力会逐步上线。

## 关于 App 与 Relayer

**Q：必须用 Hongbao 的 App 吗？**
是的。Hongbao 官方 App / Web 是唯一的标准持卡人入口。硬件、客户端、合约三者紧耦合，统一固件审计、蓝牙协议和 UX，这是产品安全与体验的基础，也是 Hongbao 与生态伙伴（钱包 / 交易所 / Dapp）合作的入口。需要品牌化展示的项目方走卡面联名定制（保留 Hongbao logo，其余区域开放，详见 [customization.md](customization.md)），不通过 App 端做定制。

**Q：必须用你们的 Relayer 吗？**
不必须。`withdraw(unlockAddress, to, v, r, s)` 没有 `msg.sender` 限制，任何 EOA 都能提交。可以用我们的（默认）、自己运营，或者让用户的 App 直接用其他钱包提交（前提是用户/你出 gas）。

**Q：用你们的 Relayer 收钱吗？**
默认我们替你出 gas，目前不收费，这是早期推广，未来可能加 fair-use 政策。如果你的发放规模大、有 SLA 要求，建议自托管 Relayer。具体见[开源仓库](https://github.com/hongbao-labs/contracts)。

**Q：能不能让用户自己出 gas？**
合约层不限制谁来提交（`withdraw` 没有 `msg.sender` 检查）。Hongbao 官方 App 默认走 Relayer 代付——这是标准持卡人体验。如果你的团队基于合约 ABI 自建客户端，可以让用户连自己的钱包付 gas（详见[开源仓库](https://github.com/hongbao-labs/contracts)），但用户体验会差一截，而且要求用户钱包里有 native 币。

## 关于损耗与售后

**Q：卡到货坏了怎么办？**
按合作合同里的 DOA 政策处理（一卡一议）。请收货时按抽样比例做验收。

**Q：用户丢卡了，能找回里面的钱吗？**
找不回（在过期前）。卡 = 私钥，丢卡 = 丢私钥 = 资产无法转出。但过期后你可以 `withdrawExpired` 把资产赎回，给用户补一张新卡。**这是我们建议的丢卡处理 SOP**：用户报损 → 你登记 → 等过期赎回 → 重发新卡 + 重新 deposit。

**Q：能否给设备序号 / 卡片防伪溯源？**
设备的 ETH 地址即唯一序列号（每张卡独立）。我们没有提供额外的防伪追踪服务——如果有需求，可以基于 ETH 地址自建 SKU 系统，定制方案请走商务对接（[contact.md](../contact.md)）。

**Q：卡的电池寿命？**
设计寿命约 3 年（深度休眠下基本不耗电；激活使用阶段消耗极低）。电池不可更换。

## 关于法务 / 合规

**Q：合规是谁的责任？**
你的责任。我们提供的是去信任的硬件加合约工具，具体派发动作（你给谁发了多少资产）由你执行，相关税务、KYC、AML、地区合规义务由你承担。合约不收手续费、不留 admin key：Hongbao 不托管、不接触你的资金，整个流程在链上、资金自始至终由你掌控。

**Q：你们会保留用户领取记录吗？**
我们的 Relayer 会有最低限度的运营日志（用于 abuse prevention 和故障排查），但不会留存超出必要的用户信息。如果你介意这部分日志，请自托管 Relayer。

**Q：如果 Hongbao 服务出问题（服务器临时不可用 / Relayer 故障），我已经发出去的卡还能领吗？**
资产层面 100% 安全。合约部署在公链上，没有 owner、不依赖任何中心化服务，资产由协议本身保证，跟我们的运营状态完全解耦。

领取层面：常态下走 Hongbao 官方 Web / App，短期故障会自动恢复。极端情况下，合约 ABI 公开、BLE 接口规范公开、CLI 工具开源，任何人都可以基于公开接口提交领取交易。懂技术的可以自行实现客户端；不会的，我们的技术支持团队随时介入。任何情况下资产都不会卡死在合约里。

## 关于产品定位

**Q：你们和硬件钱包（如市面常见的多次签名硬件钱包）是什么关系？**
完全不同的产品。硬件钱包是卖给最终用户长期管理私钥的多次签名设备，跟用户是持续关系。Hongbao 是一次性签名设备加链上分发协议，卖给资产分发方（项目方 / 交易所 / 企业），卡递到收件人手里、完成所有权转移后即作废，不是用户的“钱包”。

想给员工买个长期钱包送他，买硬件钱包。想给 1,000 名用户精确派发一笔资产并完成 onboarding，用 Hongbao。

**Q：你们和市面上的加密礼品卡产品是什么关系？**
他们是资金托管型，平台先收用户的钱，用户拿卡兑换平台的余额。Hongbao 是资金链路完全脱钩的协议，项目方自己把资产锁进合约，我们不收一分钱、不发放一分钱、不持有任何余额。架构上就不是一类产品。
